## 3) feature metric loss v2：我给你一个“不要再踩 v1 坑”的具体规格

你们 v1 的失败复盘已经很接近真相：`phi_name=depth` 很可能选错 + 对齐不一致 + 全图施加导致压死优化。

我给你 v2 的“最小可行且更稳”的技术规格（你可以直接写进 `notes/feature_loss_v2_design.md`）：

### 3.1 phi 选择：先远离 “depth/几何头”，用中间层表观特征

- **禁止默认用 depth/几何类输出当 feature loss**（它和 3DGS 的几何优化目标高度冲突，最容易把优化压成“错几何也要像 depth”）。你们 v1 已经见过这种压死模式。
- 推荐优先级（从稳到险）：
  1. **中间层 multi-scale 表观特征（stride 8/16）**
  2. 多层 concat（浅层边缘/纹理 + 中层语义）
  3. 最深层语义（最容易导致“看起来对但细节糊”，只适合做低权重正则）

> 你们不需要在文档里写“VGGT 第几层 blockX”，只要把 phi 的名字写成可复现实验配置即可（例如 `phi_name=mid_s16`）。

### 3.2 loss 形式：别用纯 L2 直接怼

- 推荐：**特征 L2-normalize 后做 cosine / (1-cosine)**，再配 Huber（或直接 cosine）
- 原因：normalize 能减小亮度/对比度差异带来的梯度发散；cosine 对幅值不敏感，更像“方向一致”。
  这会比你 v1 直接 L2 更不容易把 photometric 压死。

### 3.3 对齐：必须做“同一路径一致化检查”，否则 feature loss 在惩罚坐标误差

你们讨论包点名“最容易把 feature loss 变成惩罚坐标误差”，这是事实。

v2 必须加 2 个 sanity（写进日志/notes，不然你会 debug 一周）：

1. **GT self-consistency**：同一张 `I_gt` 走两次预处理+VGGT，loss 应接近 0（浮点误差级）。
2. **Cache round-trip**：从 cache 读出的 `phi(I_gt)` 和在线算的 `phi(I_gt)` 必须一致（否则 cache 的 meta 就是假的）。

### 3.4 施加策略：warmup + 低频 + patch + gating（你们讨论包已经写了，但必须“协议化”）

- warmup：前 20% steps 不开 feature loss（你们写了）
- low frequency：`VGGT_FEAT_EVERY = 4 或 8` 起步（你们写了）
- patch：K=32/64（你们写了）
- gating：我建议 **先用帧差 gating**，不要依赖你们当前“不稳定的 cue mining”：
  - 在 test cam 或 train cam 上，对 `I_gt(t)` 与 `I_gt(t-1)` 做 downscale 后的 abs diff，取 top‑p% 区域作为 dynamic candidates
  - patch 只从这个区域采样
  - 阈值/比例必须写死在 cfg（否则又成“挑出来的提升”）

> 这一步是关键：它让 feature loss 真正打在“动态/困难区域”，否则你大概率只在背景上做无意义正则。

### 3.5 你们现在最该补的“防守证据页”

同行那份修正方案一直抓 zero velocity（虽然打偏，但评审会问），你们讨论包也已经把“速度信号非全零”写成防守点。

我建议在 03-01 前把这页做出来并入 report-pack：

- step0/100/600 的 `||v||` 分布（全体 + dynamic region 子集）
- `||v|| < 1e-4` 比例曲线
- 以及 1 张对应的渲染现象截图（ghosting / flicker 的片段）

这页的作用：**把“zero velocity 是死路”的争论变成数据结论**。
