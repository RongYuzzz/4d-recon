## 1) Feature Loss v2：Layer 选 stride=8/16 + cosine，这条方向正确，但我要你“更保守一点”以提高 M2 成功率

### 结论拍板

- **v2 的第一版先用 stride=16 的中间层**（更抗几何/像素对齐误差，最不容易把 photometric 压死）。
- stride=8 不是不行，但**作为 v2 的第二个变体**（只有在 stride=16 跑出“趋势”后再加），否则你会在 M2 里把自己推回 v1 的坑：对齐误差 → 特征 loss 强行惩罚 → 训练变成“错方向的力”。
  你同学说 stride=8/16 取中间层是平衡，这个逻辑成立；但在你们当前阶段，“成功率”优先于“理论最优”。

### 你写的 cosine loss 实现：可用，但我补 3 个必须写死的工程细节

你给的公式：
[
L = 1 - \frac{F_{pred} \cdot F_{gt}}{|F_{pred}|\cdot|F_{gt}|+\epsilon}
]
这 OK。关键是下面 3 个点，**一个写错就等于白做**：

1. **冻结 VGGT ≠ no_grad**

- 你要的是：**VGGT 参数不更新，但梯度必须能从 feature → render image → Gaussians**。
- 所以：
  - `for p in vggt.parameters(): p.requires_grad_(False)` + `vggt.eval()`
  - **GT 特征**：`torch.no_grad()` / cache / detach 都行
  - **render 特征**：绝对不能 `torch.no_grad()`（否则 L_feat 对 Gaussians 没梯度）

1. **更稳的写法是先 normalize 再点积（避免 norm 太小）**
   建议直接用：

- `F_pred_n = F.normalize(F_pred, dim=-1, eps=1e-6)`
- `F_gt_n = F.normalize(F_gt.detach(), dim=-1, eps=1e-6)`
- `loss = 1 - (F_pred_n * F_gt_n).sum(dim=-1)`
  这样 eps 的意义更明确，梯度也更稳定。

1. **GT feature 最好“离线就归一化并落盘”**
   你们本来就要 cache GT feature。那就顺便把 `F_gt_n` 存进去：

- 训练时少一次 normalize
- 也避免你在训练里因为 dtype/autocast 搞出 GT/Pred 归一化不一致的问题

> 你提“显式写 detach 更安全”这点我同意，但别把它当核心改进。真正决定成败的是：render 分支必须保梯度 + 选更抗错配的层（stride16）。

### 为什么我反对你把“stride=8”直接当 v2 默认？

因为你们现在的渲染/几何误差在 early stage 一定存在，stride=8 的空间敏感性更强，**它更容易把 loss 变成“惩罚对齐误差”而不是“正则感知结构”**。
这就是 v1 容易出现的“PSNR 被压死”的典型机制（你们已经见过一次了）。

## 2) Gating：你提“相机运动导致帧差全图 diff”是对的，但对 SelfCap 来说大概率不用怕

SelfCap 官方描述是“长多视角视频”，`bar` 有 **18 cameras**，并且还提到同步校正列表（sync.json）——这是典型固定相机阵列采集的语气和数据形态，不像手持移动相机。([Hugging Face](https://huggingface.co/datasets/zju3dv/SelfCap-Dataset/blob/main/README.md))

所以我给你一个更硬、更工程的处理方式：

### 2.1 先做一个 30 秒的“相机是否运动”判定（比争论强）

- 解析你们 `sparse/0/images.bin`（或你们 canonical pose 源）
- 对同一 camera 的不同 frame，计算 `R,t` 的变化量
  - `max_angle_change < 1e-4 rad` 且 `max_translation < 1e-6`（量级示意）→ 当作固定机位
- 如果固定机位成立：**直接用 simple diff gating**，不要加 homography/flow

> 这是“一次性定性判定”，能把你的 gating 风险变成 0。

### 2.2 simple diff gating 的正确版本（防曝光/压缩噪声）

你担心“diff 全图都是”，在固定机位下最常见原因不是相机运动，而是：

- 自动曝光/白平衡变化
- 压缩噪声/轻微抖动

简单修正就够了（别上 homography）：

- diff 在 **灰度 + 轻微 blur** 后做
- 或做 **每帧归一化**：`I = (I - mean)/std` 再 diff
- gating 不要用“阈值”，用 **top‑p%**（例如 top 10%）选 patch，更稳

### 2.3 你提的 homography warp：我给你明确限制

- **不要默认实现**（很容易把 M1 变成工程坑）
- 只有在你真的检测到“pose 在时间上变化”时才做
- 而且 homography 对一般 3D 场景并不严格成立（除非近似平面/纯旋转），你把它当“减弱全局运动”的近似 trick 可以，但不要指望它“正确抵消背景运动”。

你写的回退逻辑是对的：**宁愿接受 gating 误判背景，也不要为了 gating 延期**。

## 4) 写作叙事“预防针”：方向对，但我不允许你写成“PSNR 下降是必要代价”这种口号

你说要把 feature loss 定义为 “Perceptual & Temporal Regularizer”，并引用 LPIPS 原文，这是正确的：LPIPS/Deep features 作为感知相似度度量的有效性是有扎实文献支撑的。([arXiv](https://arxiv.org/abs/1801.03924?utm_source=chatgpt.com))

但你写作里必须更严谨，否则评审会抓你一句话把你打穿：

### 4.1 正确表述方式：**“可观测 trade-off”**，不是“必要代价”

你应该写成：

- 我们优化的目标不是纯 MSE/PSNR，而是 `L_photo + λ L_feat (+ …)`
- `L_feat` 作为正则项，会把解推向更“结构一致/感知一致/时序更稳”的区域
- 因此可能出现：**tLPIPS 显著下降（更稳），PSNR 小幅波动（像素对齐更敏感）**
- 我们报告所有指标，并通过 λ sweep 展示 Pareto trade-off，再选 knee point

### 4.2 你要用一张图把这事“证据化”

别用嘴解释。你做一个 λ_feat sweep（很小范围就行）：

- x 轴：PSNR
- y 轴：tLPIPS
- 点：不同 λ_feat
  这张图一出来，“PSNR 小降但更稳”就从辩解变成**客观的 Pareto**。

### 4.3 你引用的两篇文献怎么放最合适

- LPIPS：用于支撑“deep features 与人类感知更一致，PSNR/SSIM 有局限”。([arXiv](https://arxiv.org/abs/1801.03924?utm_source=chatgpt.com))
- Perceptual losses (Johnson et al.)：用于支撑“用 pretrained feature 做 loss 会带来更好的视觉质量，而不是只追像素误差”。([arXiv](https://arxiv.org/abs/1603.08155?utm_source=chatgpt.com))

视频生成论文你当然也能引用，但**别堆一堆**，你毕设的主战场是 4D 重建；引用太多视频生成会让人觉得你在“借尸还魂”。
