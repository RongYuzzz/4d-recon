## C. 同行意见怎么取舍：哪些是“真问题”，哪些是“误判/跑偏”，哪些会害你

同行文档核心主张两条：

- “Zero Velocity 是死路”（认为 v=0 导致 Split4D 聚类失效）
- “KLT 只能当临时，必须恢复 VGGT 核心地位（Feature Metric Loss）并引入光流初始化”

我逐条拆解。

### 1) “Zero Velocity 是死路”——**这句话有可能是误判对象，但背后的担忧是对的**

同行引用你们的 `run_t0_zero_velocity.sh` 并据此推断“你们当前效果不佳的核心原因是 v=0”。

但从你们自己的计划文档看，`v=0` 被定义为 **T0 sanity check**（零速度退化测试），是用来验证线性运动模型能退化到静态 3DGS 的，不等价于你们正式训练时也用 v=0。
从进度文档也看不出你们正式训练是“强制 v=0”。

**所以：同行把“sanity 测试脚本”当成“主训练初始化策略”在批判，可能打错靶。**

但！他的担忧仍然成立，换个更严谨的表达是：

> 如果你们正式训练的初始 velocity 统计上“几乎全为 0 / 或者只有噪声”，那么任何依赖 `Cluster(x,v)` 的动静拆分都会退化，mask 加权也只是在做 loss reweight，不会引导出正确的运动分解。

**你们应该做的不是辩论，而是用一个统计图把这事钉死：**

- 统计 `init_points.npz` 或训练初期 Gaussians 的 `||v||` 分布（p50/p90/p99）
- 看 `||v||<eps` 的比例（例如 eps=1e-4）
- 再把“动态候选区域（由 cue 或简单帧差得到）”内的 gaussians 速度分布单独统计
  如果动态区域的速度也接近 0，那同行结论就成立：你们的运动信号没被注入。

### 2) “KLT 只能当临时”——**完全正确**

你们进度文档已经把 KLT 定位为 strong attempt，并按 stoploss 暂停大投入。
同行强调“如果把 KLT 当核心，会题文不符”，这点在答辩时确实会被攻击：题目/愿景在讲 VGGT 大模型先验，你拿 KLT 当主菜会被认为“偷换创新点”。

**建议你们统一口径：**

- KLT = “temporal correspondence baseline / sanity bridge”，仅证明端到端 loss 接入与可审计性
- 真正 Strong v2 的目标仍是 **VGGT-based cue/feature/attention**（哪怕还没完全实现，也要给出设计与失败分析证据）

### 3) “引入 RAFT / 3D scene flow 初始化 velocity”——方向对，但工程细节非常容易把你拖死

同行提法是：VGGT mask → RAFT 2D flow → “结合深度反投影”得到 3D scene flow → 写入 FreeTimeGS velocity。

这条路线最大的坑是：**深度从哪来？**

- 如果你没有可靠 depth（每帧、每像素），2D flow → 3D flow 的 lift 是不适定的。
- 你可以用渲染出来的深度去 lift，但那变成“模型用自己预测的深度生成自己的速度初始化”，容易出现自洽假象/局部最优。

**更可控的折中建议（不会把你带沟里）：**

- RAFT 不一定要做“3D velocity 初始化”，你可以先把它当成 **更强的 2D temporal correspondence extractor**（替换 KLT），用于构造 strong loss（例如 reprojection/feature consistency）
- 真正要写入 3D velocity，优先利用你们已经有的 **per-frame triangulation**（这是你们数据契约里最稀有、最值钱的东西）：用 3D 点云帧间差分/匹配得到粗 3D flow，反而比“2D flow + depth”更一致。

一句话：**别为了“v 初始化”硬上 scene flow lift，先把你手里已有的 3D 资产用到极致。**

### 4) “用 VGGT Feature Metric Loss 恢复大模型先验地位”——这是同行建议里最值钱的一条

同行给了一个极简形式：
( L_{feat} = || \text{VGGT_Feat}(I_t) - \text{VGGT_Feat}(\text{Render}(I_t)) || )。

这条路线的优势是：

- 你可以很清晰地说：**VGGT 冻结、只做前向特征、作为 perceptual prior**（叙事闭环）
- 不需要你马上把 attention 对应映射到 3D（那条最容易爆炸）

但它的工程风险也真实存在：VGGT forward 很重、训练成本会上升。你要把它做成“可控版本”，否则 GPU 时间会炸。

**可控落地版本（建议你们后续写进修正路线图）：**

- GT 图像的 VGGT features **离线预计算**（不占训练图）
- 训练时只对渲染图跑 VGGT（仍然重，但已经减半）
- 特征 loss **低频触发**（每 N step 一次）+ **低分辨率** + **patch 采样**
- 只在 mask 动态区域或高不确定区域上算（否则等于给所有像素加重 loss，容易压死 photometric）

## E. 你们下一步最该做的“诊断性动作”（不改大结构，直接定位问题根因）

> 这些动作的目标是：把“同行的批评点”从争论变成数据（证据包里可放一页图）。

### 1) velocity 是否真的“接近 0”？（直接判定同行最强指控）

输出三张统计图：

- 全体 gaussians 的 `||v||` 分布（step=0、step=100、step=600）
- 动态区域内（由 cue 或帧差得到）的 `||v||` 分布
- `||v||<eps` 的占比曲线（eps 固定）

若动态区域也接近 0：同行“zero velocity 死路”本质成立（不是指 T0 脚本，而是指主训练信号）。

### 2) cue 是否对齐错了？（你们现在最像“隐藏 bug”而不是“方法不行”）

最常见的三类坑会让 cue “看起来有”，但实际在训练里是错的：

- camera folder / frame index 对不上（cam02 的 mask 用到了 cam03）
- resize/crop 不一致（mask 与 image 尺寸/坐标系不一致）
- mask 值域/方向反了（该强调的被压低）

你们已经有 `control_weak_nocue`，这是绝佳的“验错器”：
**只要修复 cue 对齐问题，理论上 ours-weak 至少不应显著输给 control。**

### 3) weak 注入是否“太软”？（不是 cue 差，而是注入位置错）

你们当前 weak 看起来更像 “loss reweight”，而不是 “motion/disentangle guidance”。
如果要在不引入强 loss 的前提下增强弱融合的“引导力”，建议下一阶段把 mask 用在**结构性位置**上（比单纯加权 photometric 更有效）：

- 用 mask 决定哪些 gaussians 允许更大速度/更长 duration（对 static 区域施加 `||v||` 约束）
- 或用 mask 过滤/初始化动态点云（让动态分量不是从“全背景点云”里学出来的）
